## 4. Obvious Performance Bottlenecks

Ranked from most to least impactful:

### (A) Per-tile `Arrays.sort()` in `backgroundNormalize` (lines 126-134)
- **Cost:** Every tile allocates a fresh `int[count]`, copies all its pixels, and sorts it with `Arrays.sort` (dual-pivot quicksort, O(n log n)).
- **Magnitude:** For a 2480x3508 page (A4 at 300 DPI) with 64x64 tiles: ~39 x ~55 = ~2,145 tiles. Each tile has up to 4,096 pixels. Sorting 4,096 ints takes ~50k comparisons, so ~107 **million** comparisons total.
- **Why it hurts:** Sorting is unnecessary for computing a percentile — a selection algorithm (e.g., quickselect / `Arrays.parallelSort` partial sort, or a histogram-based approach) would be O(n) instead of O(n log n) per tile.

### (B) Per-pixel floating-point division in `backgroundNormalize` (line 157)
- **Cost:** `gray[idx] * 255.0 / bg` — one double division and one `Math.min` per pixel.
- **Magnitude:** 8.7 million divisions for an A4 page at 300 DPI.
- **Why it hurts:** FP division is ~10-30× slower than integer addition/multiplication. This could be replaced with integer arithmetic (`gray[idx] * 255 / (int)bg` with care to avoid overflow and preserve precision).

### (C) Worst-case O(N × R²) inpainting search loop (lines 288-302)
- **Cost:** For each foreground pixel (maybe 5-15% of page), the code searches a `(2*radius+1)²` square in scan order. With `radius=3`, that is 49 checks per foreground pixel. For ~500k foreground pixels = ~24M checks. The `(dx, dy)` loops break as soon as one background neighbor is found, which helps for thin strokes.
- **Why it hurts:** For thick text or dense regions, nearby pixels may *also* be foreground, forcing the search to scan farther. The break condition `found == 0` in both loop headers means the innermost break only exits the `dx` loop; the outer `dy` loop also checks `found == 0`, so it does stop at the first background pixel found. But if a region is entirely foreground (e.g., a photo), every pixel scans the full 7×7 window, hitting all 49 neighbors.
- **Mitigation:** A distance-transform or a 2-pass dilate-based fill would be O(N) instead of O(N × R²).

### (D) Redundant full-image pixel extractions
- `image.getRGB()` is called **three times** for the same source image:
  1. Inside `toGrayscale()` (line 76) — extracts all ARGB pixels for grayscale conversion.
  2. In `segment()` line 51 — extracts all ARGB pixels again for `origPixels` (the inpaint source).
- Each call allocates and fills an `int[width*height]`. These could be combined: the grayscale conversion could save the original ARGB array in one pass and reuse it later.

### (E) Multiple large heap allocations
Each of the following is an `int[width*height]` (for A4 at 300 DPI: ~8.7 million ints = ~35 MB each):
- `rgba` (in `toGrayscale`)
- `gray`
- `result` (in `backgroundNormalize`)
- `maskPixels`
- `origPixels`
- `resultPixels` (in `inpaintBackground`)
Plus per-tile `int[count]` arrays (up to 4096 per tile), plus the `BufferedImage` backing arrays.

Peak memory is roughly 6 × 35 MB = 210 MB for the flat arrays alone, plus the images. This can cause GC pressure on large documents.

### (F) `Math.floor` calls in per-pixel loop (lines 144-147)
- Two `Math.floor` calls per pixel. These could be replaced with integer division since `tileSize` is a power of 2 (64): `tx0 = x >> 6` (or `x / 64`).

### (G) Otsu's method is negligible
- One pass over the normalized image to build the histogram, then 256 cheap iterations. This is not a bottleneck.
